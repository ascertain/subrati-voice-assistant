### Q1: "Write a test that verifies a door lock responds within timing requirements."
```python
import pytest
import time
from unittest.mock import MagicMock

# Simulated VAS door lock interface (like our POS device communication)
class DoorLockController:
    def __init__(self, connection):
        self.conn = connection
    
    def send_unlock_command(self):
        """Send unlock signal and return response time in ms."""
        start = time.perf_counter()
        self.conn.send(b'\x01\x02\x03')  # unlock command bytes
        response = self.conn.receive(timeout=2.0)
        elapsed_ms = (time.perf_counter() - start) * 1000
        return response, elapsed_ms


# --- TEST CODE ---
@pytest.fixture
def mock_connection():
    """Mock hardware connection (like we mock POS serial connection at IKEA)."""
    conn = MagicMock()
    conn.receive.return_value = b'\x01\x00'  # ACK response
    return conn

@pytest.fixture
def door_lock(mock_connection):
    """Create controller with mocked connection."""
    return DoorLockController(mock_connection)


class TestDoorLockTiming:
    """Test door lock meets timing requirements (< 500ms response)."""
    
    MAX_RESPONSE_TIME_MS = 500  # Requirement: must respond within 500ms
    
    def test_unlock_responds_within_timing(self, door_lock):
        """Verify unlock command gets ACK within timing spec."""
        response, elapsed_ms = door_lock.send_unlock_command()
        
        assert response == b'\x01\x00', f"Expected ACK, got {response}"
        assert elapsed_ms < self.MAX_RESPONSE_TIME_MS, \
            f"Response too slow: {elapsed_ms:.1f}ms > {self.MAX_RESPONSE_TIME_MS}ms"
    
    def test_unlock_handles_timeout(self, mock_connection):
        """Verify graceful handling when hardware doesn't respond."""
        mock_connection.receive.side_effect = TimeoutError("No response")
        lock = DoorLockController(mock_connection)
        
        with pytest.raises(TimeoutError):
            lock.send_unlock_command()
    
    def test_unlock_retries_on_nack(self, mock_connection):
        """Verify retry logic on NACK (like POS retry on card read failure)."""
        # First call: NACK, second call: ACK
        mock_connection.receive.side_effect = [b'\x01\x01', b'\x01\x00']
        lock = DoorLockController(mock_connection)
        
        # Should retry and succeed
        response, _ = lock.send_unlock_command()
        assert mock_connection.send.call_count >= 1
```

## Predicted Questions & Answers:

### Q1: "Explain the difference between SIL and HIL testing. When do you use each?"
**Answer:** "Think of it like testing a self-checkout kiosk at IKEA:
- **SIL (Software-in-the-Loop):** I run the kiosk software on my laptop with a simulated barcode scanner and payment terminal. Fast, cheap, catches logic bugs. For VAS: run door lock firmware in a simulator, mock the BLE/NFC signals.
- **HIL (Hardware-in-the-Loop):** I connect the actual kiosk hardware (real scanner, real POS terminal) to a test controller that simulates customer actions. Catches timing, electrical, and integration bugs. For VAS: real ECU on a bench, real BLE antenna, test controller sends simulated key signals.

**When to use:** SIL first (fast feedback, catches 70% of bugs), then HIL (catches timing/hardware bugs), then vehicle (catches system-level integration). Each level catches different bug types."

### Q2: "Tell me about a time you debugged a hardware-software interface issue."
**Answer:** "At IKEA, our Morpho biometric devices randomly failed fingerprint capture. Software logs showed 'timeout' but the device wasn't crashing. 

My debugging process:
1. **Reproduced:** Only happened under load (multiple transactions)
2. **Instrumented:** Added USB communication timing logs
3. **Found pattern:** Failures clustered after 15+ minutes of continuous use
4. **Root cause:** USB buffer overflow — the device firmware sent data faster than our Python driver consumed it during peak usage
5. **Fix:** Implemented flow control in the driver with a read buffer, and added a hardware watchdog reset command

Cross-team impact: I documented the issue, worked with the Morpho vendor to confirm it was a known firmware limitation, and updated our test suite to include 'sustained load' tests that ran for 30+ minutes."

### Q3: "How would you design a test strategy for a new VAS feature like Phone-as-Key?"
**Answer:** "I'd break it into layers — same way I designed the IKEA mobile payment testing:

**Requirements:** Phone connects via BLE → car authenticates → unlocks within 1 second

**Test levels:**
1. **Unit tests:** Crypto authentication logic, BLE protocol parsing (Python mocks)
2. **Integration (SIL):** Firmware + BLE stack in simulator — does handshake work?
3. **HIL:** Real BLE hardware + real ECU — test timing, signal strength, interference
4. **System/Vehicle:** Real phone + real car — walk-up scenario, edge cases

**Edge cases I'd prioritize:**
- Phone battery low (weak BLE signal)
- Multiple phones nearby (which one to authenticate?)
- Walk-away lock timing
- Relay attack prevention
- Firmware update during active connection

**Automation approach:** CI runs SIL tests on every commit, HIL runs nightly, vehicle tests run weekly."

### Q4: "How do you handle testing when hardware isn't available?"
**Answer:** "This is exactly what we faced at IKEA — we couldn't have POS terminals on every developer's desk. Our approach:
1. **Built simulators:** Python classes that mimicked POS serial protocol responses (configurable: happy path, timeout, error codes)
2. **Record-replay:** Captured real hardware communication sessions, replayed in tests
3. **Contract testing:** Defined interface contracts — if simulator passes, real hardware should too
4. **Gradually elevated:** Dev → SIL (simulated) → HIL (real hardware, nightly) → Store (vehicle equivalent)

Key insight: the simulator should be 'good enough' to catch logic bugs but NOT so perfect that it hides hardware quirks. I always keep one HIL setup that runs against real hardware nightly."

### Q1: "Design a test system for Vehicle Access Systems from scratch."
**Answer:** "Let me break this into components:

**System Under Test:** ECU running firmware that controls door locks via BLE/NFC/UWB

**My test architecture:**
```
┌─────────────────────────────────┐
│       CI/CD Pipeline            │
├─────────────────────────────────┤
│  Level 1: Unit Tests            │  ← Every commit (seconds)
│  Python + C unit tests          │
├─────────────────────────────────┤
│  Level 2: SIL Tests             │  ← Every PR (minutes)
│  Firmware simulator + Python    │
│  test harness                   │
├─────────────────────────────────┤
│  Level 3: HIL Tests             │  ← Nightly (hours)
│  Real ECU + test controller     │
│  + signal generators            │
├─────────────────────────────────┤
│  Level 4: Vehicle Tests         │  ← Weekly/Release
│  Real car + test phones/fobs    │
└─────────────────────────────────┘
```

**Key design decisions:**
1. **Same test cases, different fixtures** — a test like 'unlock within 500ms' runs at SIL (fast) AND HIL (real) using pytest fixture swapping
2. **Observability built in** — structured JSON logs from firmware, timing traces from test harness, CAN bus captures
3. **Failure classification** — each test failure auto-categorized: timing, protocol, hardware, environment
4. **Flakiness quarantine** — flaky tests auto-moved to separate suite, investigated weekly"

### Q2: "How would you make the system observable and diagnosable?"
**Answer:** "From my IKEA experience debugging POS issues remotely in stores:

1. **Structured logging:** Every firmware event → JSON with timestamp, event_type, state_before, state_after
2. **State machine visualization:** VAS firmware is essentially a state machine. Log every transition with trigger.
3. **Timing traces:** Instrument entry/exit of every critical path. Can see: 'BLE connect took 340ms, auth took 50ms, actuator took 110ms = 500ms total'
4. **Test artifacts:** Every test run saves: logs, CAN traces, BLE captures, timing diagrams — auto-attached to failure reports
5. **Replay capability:** Capture the exact inputs (BLE signals, CAN messages) so a failure can be replayed in SIL

The principle: **if it fails in the field, I should be able to reproduce it on my desk within 30 minutes.**"

### Q3: "How do you decide what to automate first?"
**Answer:** "Risk × Frequency × Cost matrix:
- **High risk + runs often + expensive to test manually** = Automate immediately (e.g., door unlock timing)
- **High risk + rare** = Automate for regression, but also manual exploratory (e.g., relay attacks)
- **Low risk + runs often** = Automate for CI speed (e.g., BLE protocol compliance)
- **Low risk + rare** = Maybe don't automate (e.g., obscure edge case in legacy feature)

At IKEA: first automated the checkout flow (ran 1000x/day, high revenue risk), then payment edge cases, then kiosk hardware interactions."

### Q4: "Walk me through how you'd investigate a field failure: 'Customer reports door didn't unlock.'"
**Answer:** "My investigation playbook (same one I used for IKEA store-reported POS failures):

**Step 1: Gather evidence** (5 min)
- When exactly? (timestamp → pull logs)
- Which vehicle? (VIN → pull ECU firmware version)
- Which key method? (phone/fob/NFC)
- Weather/environment?

**Step 2: Check known patterns** (5 min)
- Is this the same firmware version where we saw BLE timeout issues?
- Is this a known flaky hardware batch?
- Any recent OTA updates?

**Step 3: Reproduce** (30 min)
- Pull the ECU logs from that time window
- Look at BLE connection attempt: did phone connect? did auth pass? did actuator fire?
- Find exactly WHERE in the state machine it got stuck

**Step 4: Narrow root cause** (varies)
- If BLE didn't connect → signal strength? interference? phone OS version?
- If auth failed → crypto key rotation issue? timing?
- If actuator didn't fire → CAN message sent but not received? hardware fault?

**Step 5: Fix and prevent**
- Fix the bug
- Add a test that catches THIS specific failure
- Ask: could we have caught this at SIL/HIL? If yes, add it there too."
