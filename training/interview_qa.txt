Q: Write a Python test for a BLE device connection with timeout handling.

A:
```python
import pytest
import time
from unittest.mock import MagicMock, patch

class BLEDevice:
    """Represents a BLE connection to VAS hardware (like Morpho USB connection)."""
    
    def __init__(self, adapter):
        self.adapter = adapter
        self.connected = False
    
    def connect(self, mac_address, timeout=5.0):
        """Connect to BLE device within timeout."""
        start = time.perf_counter()
        while time.perf_counter() - start < timeout:
            result = self.adapter.scan_and_connect(mac_address)
            if result == "connected":
                self.connected = True
                return True
            time.sleep(0.1)
        raise TimeoutError(f"Failed to connect to {mac_address} within {timeout}s")
    
    def send_command(self, cmd_bytes):
        """Send command and return response."""
        if not self.connected:
            raise ConnectionError("Not connected")
        return self.adapter.write_read(cmd_bytes)


# --- TESTS ---

@pytest.fixture
def mock_adapter():
    """Mock BLE adapter (like we mock Morpho device at IKEA)."""
    return MagicMock()

@pytest.fixture
def ble_device(mock_adapter):
    return BLEDevice(mock_adapter)


class TestBLEConnection:
    
    def test_connect_success(self, ble_device, mock_adapter):
        """Happy path: device connects immediately."""
        mock_adapter.scan_and_connect.return_value = "connected"
        
        result = ble_device.connect("AA:BB:CC:DD:EE:FF")
        
        assert result is True
        assert ble_device.connected is True
        mock_adapter.scan_and_connect.assert_called_with("AA:BB:CC:DD:EE:FF")
    
    def test_connect_timeout(self, ble_device, mock_adapter):
        """Device doesn't respond — should raise TimeoutError."""
        mock_adapter.scan_and_connect.return_value = "not_found"
        
        with pytest.raises(TimeoutError):
            ble_device.connect("AA:BB:CC:DD:EE:FF", timeout=0.5)
    
    def test_connect_retry_then_success(self, ble_device, mock_adapter):
        """Device found after retries (like Morpho needing warm-up)."""
        mock_adapter.scan_and_connect.side_effect = [
            "not_found", "not_found", "connected"
        ]
        
        result = ble_device.connect("AA:BB:CC:DD:EE:FF", timeout=5.0)
        
        assert result is True
        assert mock_adapter.scan_and_connect.call_count >= 3
    
    def test_send_command_when_not_connected(self, ble_device):
        """Should fail gracefully if not connected."""
        with pytest.raises(ConnectionError, match="Not connected"):
            ble_device.send_command(b'\x01\x02')
    
    def test_send_unlock_command(self, ble_device, mock_adapter):
        """Send unlock and verify response."""
        mock_adapter.scan_and_connect.return_value = "connected"
        mock_adapter.write_read.return_value = b'\x00\x01'  # ACK
        
        ble_device.connect("AA:BB:CC:DD:EE:FF")
        response = ble_device.send_command(b'\x01\x00\x55')  # Unlock cmd
        
        assert response == b'\x00\x01'
```

Q: Write a function that validates CAN bus message timing.

A:
```python
from dataclasses import dataclass
from typing import List

@dataclass
class CANMessage:
    timestamp_ms: float
    message_id: int
    data: bytes

def validate_message_timing(
    messages: List[CANMessage],
    expected_id: int,
    min_interval_ms: float,
    max_interval_ms: float
) -> dict:
    """
    Validate that CAN messages arrive at expected intervals.
    Like validating POS heartbeat messages at IKEA.
    
    Returns: {"valid": bool, "violations": [...], "stats": {...}}
    """
    filtered = [m for m in messages if m.message_id == expected_id]
    
    if len(filtered) < 2:
        return {"valid": False, "violations": ["Not enough messages"], "stats": {}}
    
    intervals = []
    violations = []
    
    for i in range(1, len(filtered)):
        interval = filtered[i].timestamp_ms - filtered[i-1].timestamp_ms
        intervals.append(interval)
        
        if interval < min_interval_ms:
            violations.append(
                f"Message {i}: interval {interval:.1f}ms < min {min_interval_ms}ms"
            )
        elif interval > max_interval_ms:
            violations.append(
                f"Message {i}: interval {interval:.1f}ms > max {max_interval_ms}ms"
            )
    
    stats = {
        "count": len(filtered),
        "avg_interval_ms": sum(intervals) / len(intervals),
        "min_interval_ms": min(intervals),
        "max_interval_ms": max(intervals),
    }
    
    return {
        "valid": len(violations) == 0,
        "violations": violations,
        "stats": stats,
    }


# Test for the validator:
def test_can_timing_valid():
    messages = [
        CANMessage(0, 0x100, b'\x01'),
        CANMessage(100, 0x100, b'\x02'),
        CANMessage(200, 0x100, b'\x03'),
        CANMessage(300, 0x100, b'\x04'),
    ]
    result = validate_message_timing(messages, 0x100, min_interval_ms=80, max_interval_ms=120)
    assert result["valid"] is True

def test_can_timing_violation():
    messages = [
        CANMessage(0, 0x100, b'\x01'),
        CANMessage(50, 0x100, b'\x02'),   # Too fast!
        CANMessage(200, 0x100, b'\x03'),
    ]
    result = validate_message_timing(messages, 0x100, min_interval_ms=80, max_interval_ms=120)
    assert result["valid"] is False
    assert len(result["violations"]) > 0
```

Q: How do you make tests reliable in CI when they depend on hardware timing?

A: "At IKEA, this was our biggest challenge with POS device tests. Here's what I learned:

1. **Separate timing tests from logic tests** — logic tests use mocks (always fast), timing tests only run on dedicated hardware
2. **Use configurable timeouts** — not hardcoded values:
```python
TIMEOUT = float(os.environ.get('BLE_TIMEOUT', '2.0'))  # Longer in CI
```
3. **Retry with backoff for hardware operations** (not for assertions):
```python
@retry(attempts=3, delay=0.5, exceptions=(ConnectionError,))
def connect_device(adapter, mac):
    return adapter.connect(mac)
```
4. **Report timing as metrics** — even passing tests report their timing so we spot degradation early
5. **Flakiness budget** — if a test fails > 2% of runs, it goes into quarantine for investigation"

```
tests/
├── conftest.py              # Shared fixtures, parametrize
├── unit/                    # Fast, no hardware — 80% of tests
│   ├── test_auth_logic.py

│   └── test_signal_strength.py

    ├── simulators.py        # Hardware simulators
    ├── assertions.py        # Custom assertions (timing, protocol)
    └── retry.py             # Retry utilities
```

Key rules:
1. **One assertion per test** — easy to see what broke
2. **Fixtures manage state** — tests never call setup/teardown directly
3. **No sleep in tests** — use polling or events
4. **Custom assertions for domain** — `assert_responded_within(500)` not `assert time < 0.5`
5. **Tests tell a story** — name = scenario, not implementation"

Q: Explain SIL vs HIL — when do you use each?

A: "I think of it like testing levels for an IKEA self-checkout kiosk:

**SIL (Software-in-the-Loop):**
- Run the firmware logic on a PC, simulated inputs/outputs
- Fast: seconds to run, everyone can run it locally
- Catches: logic bugs, state machine errors, protocol parsing
- Misses: timing, electrical, real hardware quirks
- Example: test that BLE handshake logic produces correct auth token

**HIL (Hardware-in-the-Loop):**
- Real ECU on a bench, real sensors, test controller as stimulus
- Slower: minutes, needs dedicated bench
- Catches: timing violations, signal integrity, firmware-hardware mismatch
- Misses: full vehicle integration, multi-ECU interactions
- Example: test that door actually unlocks within 500ms when real BLE signal received

**My rule of thumb:**
- If it's a logic question → SIL (fast feedback)
- If it involves timing, signals, or real interfaces → HIL
- If it involves multiple systems together → vehicle/system test

At IKEA: we ran POS protocol tests in SIL (software simulator), then validated real terminal timing on a bench (our version of HIL), then final validation in a pilot store."

Q: Walk me through debugging a failure where the car door doesn't unlock via phone.

A: "I'll use the same systematic approach I use for IKEA POS failures:

**Step 1: Scope the problem (2 min)**
- Is it one car or many? (hardware batch vs software bug)
- Is it one phone model or all? (compatibility vs protocol)
- Was there a recent OTA update? (regression)
- Does key fob work? (isolates BLE vs lock mechanism)

**Step 2: Trace the signal path (5 min)**
Look at each link in the chain:
```
Phone App → BLE signal → Antenna → ECU receives → Auth check → CAN message → Actuator → Door unlocks
```
Which link is broken?

**Step 3: Check logs and data (10 min)**
- ECU log: did it even receive a BLE advertisement?
- Auth log: did crypto handshake complete?
- CAN trace: was unlock command sent on the bus?
- Actuator log: did motor engage?

**Step 4: Isolate (15 min)**
If BLE connection fails:
- Signal strength? (RSSI too low = antenna issue)
- Phone BLE version compatible?
- Firmware state? (is ECU in sleep mode not scanning?)

If auth fails:
- Key rotation? (expired crypto key)
- Clock sync? (timestamp validation failing)

If CAN message sent but door doesn't unlock:
- CAN bus issue? (check error frames)
- Actuator mechanical failure?
- Power issue? (battery voltage too low for actuator)

**Step 5: Reproduce and fix**
- Write a test that recreates the exact failure condition
- Fix the bug
- Add the test to HIL regression suite so it never escapes again

At IKEA, I used this exact approach when Morpho fingerprint readers randomly timed out. Turned out to be USB buffer overflow under sustained load. I added a load-duration test to catch it."

Q: How would you test a feature that has security implications (e.g., relay attack prevention)?

A: "Security testing needs a different mindset — adversarial, not just functional.

For VAS relay attack prevention:
1. **Understand the attack:** Thief uses two devices to relay BLE signal from your phone (inside house) to car (in driveway)

- **Adversarial test:** Actually attempt relay attack with real equipment — verify car stays locked

- SIL relay tests run on every commit (fast, catches regressions)
- HIL adversarial tests run weekly (more realistic)
- Penetration test by security team quarterly (human creativity)

Q: How do you ensure test coverage for state machines in embedded systems?

A: "Door lock is a perfect state machine example:

```
States: Locked, Unlocked, Error, Sleep
Transitions: unlock_cmd, lock_cmd, timeout, fault, wake

       ┌─── unlock ───┐
       ▼               │
   [Locked] ──────► [Unlocked]
       │                │
       │   timeout      │
       ▼                ▼
    [Sleep]         [Locked]
       │
    fault
       ▼
    [Error]
```

**My coverage strategy:**

1. **All valid transitions** (happy paths):
   - Locked → Unlocked (via unlock command)
   - Unlocked → Locked (via lock command or timeout)

2. **All invalid transitions** (boundary cases):
   - Locked → Locked (double lock — should it error or ignore?)
   - Error → any (what recovers from error?)

3. **Edge cases in transitions:**
   - Command received during state transition (race condition)
   - Power loss during transition
   - Multiple simultaneous commands

4. **Long-running state scenarios:**
   - Stay in unlocked for hours (does timeout work?)
   - Rapid lock/unlock cycles (stress test)

**Tool:** I generate test cases from a state machine model. Every state + every event = one test. If there are N states and M events, I need at minimum N×M tests to cover all transitions (valid and invalid)."

Q: How do you handle testing across multiple ECUs that need to coordinate?

A: "At IKEA, this is like testing the checkout flow where POS terminal, receipt printer, and inventory system must coordinate.

For VAS multi-ECU: phone talks to Body Controller → Body Controller sends CAN to Door Module → Door Module drives actuator.

**Testing layers:**
1. **Interface contracts** — define exactly what messages each ECU expects/produces. Test each in isolation against the contract.
2. **Integration (SIL)** — run all ECU software in simulation, verify end-to-end message flow
3. **Integration (HIL)** — real ECUs on bench, verify timing and signal integrity
4. **Fault injection** — what if one ECU is slow? What if a message is lost?

**Key pattern — Contract Testing:**
```
ECU_A produces: CAN ID 0x100, 8 bytes, every 100ms
ECU_B expects:  CAN ID 0x100, 8 bytes, within 150ms

Test A independently: does it produce correctly?
Test B independently: does it handle correctly when received?
Integration test: does the actual timing work?
```

At IKEA: we used this for POS ↔ backend ↔ inventory integration. Each team tests against the contract. Integration tests verify the contracts work together."

```
│  - Test reports → dashboard             │
└─────────────────────────────────────────┘
```

**Key design decision:** Same test code runs at all levels. Only the fixture changes:
```python
@pytest.fixture
def door_lock(request):
    env = request.config.getoption('--env')
    if env == 'mock':
        return MockDoorLock()
    elif env == 'sil':
        return SILDoorLock(qemu_instance)
    elif env == 'hil':
        return HILDoorLock(serial_port='/dev/ttyUSB0')
```

This means: write once, test at every level. Maximum reuse."

Q: How would you design for observability in an embedded system?

A: "From my experience debugging POS device issues in IKEA stores remotely — observability is the difference between 5-minute and 5-day debugging.

**My observability framework for VAS:**

**1. Structured Event Logging (in firmware):**
```c
// Every state change logged as structured event
LOG_EVENT(EVT_BLE_CONNECT, .mac=peer_mac, .rssi=-45, .latency_ms=120);
LOG_EVENT(EVT_AUTH_START, .method=PHONE_KEY, .key_id=0x1234);
LOG_EVENT(EVT_AUTH_RESULT, .success=true, .duration_ms=50);
LOG_EVENT(EVT_DOOR_UNLOCK, .source=BLE, .actuator_current_ma=150);
```

**2. Timing Traces:**
Every critical path has entry/exit timestamps:
```
[T+0ms]   BLE advertisement received
[T+120ms] Connection established
[T+170ms] Auth challenge sent
[T+220ms] Auth response received
[T+225ms] Auth verified
[T+230ms] CAN unlock command sent
[T+235ms] Actuator ACK received
[T+340ms] Door unlocked (total: 340ms ✓ < 500ms requirement)
```

**3. Test Artifact Collection:**
Every test failure auto-captures:
- ECU log (last 1000 events)
- CAN bus trace (last 5 seconds)
- BLE sniffer capture
- Power rail measurements
- Screenshot of state machine position

**4. Dashboard:**
- Real-time: test pass rate, average timing, flakiness score
- Trend: are tests getting slower? Are new bugs appearing?
- Drill-down: click a failure → see all artifacts

**Principle:** If someone reports 'door didn't unlock,' I should be able to find the root cause from logs alone within 30 minutes."

Q: How would you handle a situation where SIL tests pass but HIL tests fail?

A: "This is the most interesting debugging scenario! It means the gap between simulation and reality.

**Common root causes (from my experience):**

1. **Timing:** SIL runs on fast PC, HIL runs on real MCU clock. Something is timing-sensitive.
   - Fix: add timing constraints to SIL simulator

2. **Signal fidelity:** SIL gives perfect signals, HIL has noise/interference.
   - Fix: add noise model to SIL

3. **Initialization sequence:** SIL starts in known state, HIL has boot sequence.
   - Fix: HIL fixture must wait for boot + known state

4. **Hardware quirks:** real device behaves slightly different from datasheet.
   - Fix: update simulator to match real behavior (document the quirk!)

5. **Environment:** temperature, power supply variation, EMI.
   - Fix: these can't be in SIL — accept some tests are HIL-only

**My process:**
1. Get the exact failure from HIL (log, trace)
2. Try to reproduce in SIL with more realistic parameters
3. If I can reproduce in SIL → fix the simulator fidelity → fix the bug
4. If I can't reproduce in SIL → it's a hardware/environment issue → keep as HIL-only test

**Key insight:** SIL-HIL gap is a SIGNAL. It tells you what your simulator doesn't model yet. Reduce the gap over time."

Q: How do you approach CI/CD for embedded systems where you need real hardware?

A: "This is a real constraint. At IKEA we solved it for POS devices:

**Three-tier CI:**

| Tier | Trigger | Duration | Hardware | What it catches |
|------|---------|----------|----------|-----------------|
| Fast CI | Every commit | < 5 min | None (mocks) | Logic bugs, API breaks |
| SIL CI | Every PR merge | < 30 min | Simulators | Integration, protocol |
| HIL CI | Nightly + release | < 2 hours | Real ECUs | Timing, hardware |

**Hardware farm management:**
- Pool of HIL benches (like a device farm)
- Tests queue up, get assigned to next available bench
- Each bench has known configuration (firmware version, calibration)
- Auto-reset between test runs (power cycle + flash baseline firmware)

**When HIL is busy/broken:**
- SIL results are 'go/no-go' for merge
- HIL failures are next-day priority investigation
- Release requires clean HIL run (gate)

**At IKEA:** We had 5 POS test terminals that CI scheduled against. If one broke, tests distributed to others. Same concept for a VAS HIL farm."

Q: Design a root cause analysis process for recurring field failures.

A: "I'd implement what I call 'Failure Funnel':

**Level 1: Detection (automated)**
- Telemetry from vehicles detects anomaly (door unlock > 2 seconds or failure)
- Auto-creates ticket with: VIN, timestamp, firmware version, ECU logs

**Level 2: Classification (semi-automated)**
- Cluster similar failures (same firmware? same phone model? same region?)
- Match against known issues database
- Priority: Safety (P1) → Function (P2) → Convenience (P3)

**Level 3: Investigation (engineer)**
- Pull detailed logs from affected vehicle
- Trace signal path: where did it break?
- Check: is this new or regression?

**Level 4: Reproduce**
- Attempt to reproduce on HIL bench with same firmware + conditions
- If reproducible → fix → add regression test
- If not reproducible → add more logging → wait for next occurrence

**Level 5: Prevention**
- Root cause fixed
- Test added at lowest possible level (SIL if possible)
- Simulator updated if it missed this scenario
- Knowledge shared: 'lessons learned' document

**Metrics:**
- Mean time to detect (MTTD)
- Mean time to reproduce (MTTR)
- Mean time to fix (MTTF)
- Escape rate per test level (where SHOULD this have been caught?)

At IKEA: we used this for POS failure analysis. Tracked which test level should have caught each production bug, then invested in closing those gaps."
